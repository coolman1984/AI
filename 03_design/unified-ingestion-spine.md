# Unified Ingestion Spine

## الهدف

بناء عمود موحد لاستخراج البيانات من المصادر الموعودة داخل المشروع:
- Excel و CSV
- جداول PDF
- PowerPoint
- البريد الإلكتروني ومرفقاته

ثم إدخالها إلى تخزين محلي منظم، بحيث يصبح الاستعلام والحسابات الدقيقة ممكنين فوق نفس المسار بدون اختراع منطق جديد لكل مصدر.

## القاعدة الحاكمة

- الاستخراج لا يخترع أرقامًا.
- كل قيمة تدخل التخزين يجب أن تكون ناتجة من parsing صريح.
- كل صف غير موثوق أو غير قابل للقراءة يذهب إلى الرفض مع سبب واضح.
- الحسابات النهائية لا تأتي من النص الخام، بل من طبقة الاستعلام والحساب الموثوقة.

## المسار الموحد

كل مصدر يمر بنفس السلم:
1. تحديد المصدر
2. استخراج الصفوف أو الحقول
3. استنتاج النوع
4. التحقق أو الرفض
5. الإدخال إلى التخزين المحلي
6. الاستعلام والحساب من الطبقة الموثوقة

## ما الذي نريد تجنبه

نرفض تصميم التخزين الذي يعتمد على:
- أعمدة عامة من نوع `col_1`, `col_2`, `col_3`
- تخزين كل شيء كنص فقط
- إجبار طبقة الحساب على cast عشوائي وقت التشغيل

هذا يخلق التباسًا، ويكسر الحسابات الدقيقة، ويجعل كل استعلام هشًا.

## نموذج التخزين المقترح

نموذج التخزين يجب أن يحفظ ثلاثة أشياء معًا:
- سياق عملية الاستخراج
- الصف المستخرج نفسه
- الحقول الدلالية داخل الصف مع نوعها وقيمتها الجاهزة للحساب

## الجداول المقترحة

```sql
CREATE TABLE extraction_runs (
    id            BIGINT PRIMARY KEY,
    source_type   VARCHAR NOT NULL,
    source_file   VARCHAR NOT NULL,
    source_sheet  VARCHAR,
    rows_loaded   INTEGER NOT NULL,
    rows_rejected INTEGER NOT NULL DEFAULT 0,
    ingested_at   TIMESTAMP DEFAULT now()
);

CREATE TABLE extracted_rows (
    id            BIGINT PRIMARY KEY,
    run_id        BIGINT NOT NULL REFERENCES extraction_runs(id),
    row_index     INTEGER NOT NULL,
    source_file   VARCHAR NOT NULL,
    source_type   VARCHAR NOT NULL,
    row_hash      VARCHAR NOT NULL,
    ingested_at   TIMESTAMP DEFAULT now(),
    UNIQUE(source_file, source_type, row_hash)
);

CREATE TABLE extracted_fields (
    row_id        BIGINT NOT NULL REFERENCES extracted_rows(id),
    field_name    VARCHAR NOT NULL,
    field_value   VARCHAR NOT NULL,
    field_type    VARCHAR NOT NULL,
    typed_value   DOUBLE,
    date_value    DATE,
    PRIMARY KEY (row_id, field_name)
);

CREATE TABLE rejected_rows (
    id            BIGINT PRIMARY KEY,
    run_id        BIGINT NOT NULL REFERENCES extraction_runs(id),
    source_file   VARCHAR NOT NULL,
    source_type   VARCHAR NOT NULL,
    row_raw       VARCHAR NOT NULL,
    reject_reason VARCHAR NOT NULL,
    ingested_at   TIMESTAMP DEFAULT now()
);
```

## لماذا هذا أفضل

- اسم الحقل محفوظ دلاليًا بدل رقم عمود غامض
- القيمة الخام تظل موجودة للمراجعة
- القيمة الرقمية الجاهزة للحساب متاحة مباشرة
- التاريخ له خانة واضحة
- الصفوف المرفوضة لا تختفي
- نفس النموذج يمكنه خدمة الإكسيل، والـ PDF، والـ PowerPoint، والبريد

## ترتيب التنفيذ

### IS1
تعريف عقد التخزين نفسه، ومراجعة أسمائه، وحدود الأنواع، ومنطق الرفض.

### IS2
تقوية مسار Excel و CSV:
- تعدد الشيتات
- استنتاج الأنواع
- صفوف الإجماليات
- التكرار
- القيم غير القابلة للتحويل

### IS3
إضافة استخراج جداول PDF:
- جدول واضح
- صفوف قابلة للتطبيع
- ثقة منخفضة تذهب للمراجعة

### IS4
إضافة استخراج PowerPoint:
- نصوص الشرائح
- جداول الشرائح
- حفظ رقم الشريحة ضمن السياق

**ملاحظة بيانات الاختبار:**
يتم توليد ملفات PowerPoint صناعية داخل الاختبارات أولًا.
ثم تضاف ملفات حقيقية لاحقًا عند توفرها.

### IS5
إضافة استخراج البريد:
- جسم الرسالة
- العنوان
- المرسل
- التاريخ
- المرفقات
- توجيه المرفقات إلى المستخرج المناسب

**ملاحظة بيانات الاختبار:**
يتم توليد رسائل صناعية داخل الاختبارات أولًا.
ثم تضاف عينات حقيقية لاحقًا عند توفرها.

### IS6
إثبات أن التخزين الناتج يدعم:
- Query موحد
- حسابات دقيقة
- تجميعات سليمة
- عدم خلط النصوص بالقيم الرقمية

## أسئلة الاستعلام التي يجب أن يقدر عليها المسار

- ما كل الصفوف المستخرجة من ملف معين؟
- ما كل القيم الخاصة بحقل معين عبر عدة مصادر؟
- ما الصفوف المرفوضة ولماذا؟
- ما القيم الرقمية الجاهزة للحساب داخل فترة معينة؟
- ما البيانات التي دخلت من البريد ثم انتهت إلى حساب تكلفة أو مقارنة؟

## قواعد الاختبار

لكل مرحلة من IS1 إلى IS6:
1. نكتب الاختبار أولًا
2. نجعل الاختبار يفشل بوضوح
3. نبني التنفيذ
4. نعيد الاختبار المستهدف
5. ثم نعيد البوابة الأوسع

## بوابة الانتقال لما بعده

لا ننتقل إلى البنية الثقيلة قبل تحقق الآتي:
- وصول ملف حقيقي واحد على الأقل
- إثبات التخزين الموحد
- نجاح مسارين حقيقيين على الأقل فوق CSV
- إمكان تنفيذ استعلامات وحسابات دقيقة من التخزين نفسه

## النتيجة المطلوبة

الهدف ليس مجرد قراءة ملفات أكثر.

الهدف هو:
**مسار موحد يجعل كل مصدر قابلًا للاستخراج والتخزين والاستعلام والحساب بثقة، بدل حلول منفصلة وهشة لكل نوع ملف.**
